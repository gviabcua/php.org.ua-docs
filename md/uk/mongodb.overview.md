# Огляд архітектури

У цьому розділі пояснюється, як різні частини драйвера з'єднуються разом. Від різних мов виконання, через модуль та до бібліотек PHP над ними. Ця нова архітектура замінила старий модуль `mongo`. Ми називаємо новий модуль `mongodb`

![Схема архітектури](images/f3bc48edf40d5e3e09a166c7fadc7efb-driver_arch.png)

У верхній частині цього стека знаходиться [» библиотека PHP](https://github.com/mongodb/mongo-php-library), яку ми розповсюджуватимемо у вигляді пакета Composer. Ця бібліотека надасть API, аналогічний тому, що користувачі очікують від старого драйвера mongo (наприклад, методи CRUD, об'єкти баз даних та колекцій, помічники команд), і ми очікуємо, що він буде загальною залежністю для більшості програм, створених з MongoDB. Ця бібліотека також реалізовуватиме спільні [» специфікації](https://github.com/mongodb/specifications) в інтересах покращення узгодженості API з усіма [» драйверами](https://www.mongodb.com/docs/drivers/), що підтримуються MongoDB (і, сподіваємося, також деякими драйверами спільноти).

Під цією бібліотекою у нас є драйвер нижнього рівня. Цей модуль буде ефективно пов'язувати PHP та наші системні бібліотеки ([» libmongoc](https://github.com/mongodb/mongo-c-driver) і [» libbson](https://github.com/mongodb/mongo-c-driver/tree/master/src/libbson)). Цей модуль надаватиме ідентичний загальнодоступний API для найбільш важливої ​​та чутливої ​​до продуктивності функціональності:

-   Управління з'єднанням
-   BSON кодування та декодування
-   Серіалізація документа об'єкта (для підтримки бібліотек ODM)
-   Виконання команд та операцій запису
-   Обробка запитів та курсорів

Роз'єднуючи внутрішні компоненти драйвера та високорівневий API у модулях та бібліотеках PHP, відповідно, ми сподіваємося зменшити навантаження на обслуговування та прискорити ітерацію нових функцій. Як приємний побічний ефект, це також полегшить будь-кому робити свій внесок у драйвера. Крім того, ідентичний загальнодоступний API значно спростить портування програми на середовища виконання PHP, незалежно від того, чи використовує програму драйвер низького рівня безпосередньо чи бібліотеку PHP вищого рівня.

[» GridFS](https://www.mongodb.com/docs/manual/core/gridfs/) - чудовий приклад того, чому ми вибрали цей напрямок. Хоча ми реалізували GridFS у C для нашого старого драйвера mongo, насправді це є досить високорівнева специфікація. Його API - це просто абстракція для доступу до двох колекцій: файлів (тобто метаданих) та частин (тобто блоків даних). Аналогічно, весь синтаксичний цукор, знайдений у старому драйвері mongo, такий як обробка завантажених файлів або представлення файлів GridFS у вигляді потоків PHP, може бути реалізований на чистому PHP. За умови, що у нас є ефективні методи читання та запису в колекції GridFS і завдяки нашим низькорівневим модулям ми будемо перекладати цей API на PHP - безпрограшний варіант.

Раніше я згадував, що ми очікуємо, що бібліотека PHP буде загальною залежністю для *більшості* додатків, але не для *всіх*. Деякі користувачі можуть віддати перевагу API-інтерфейсу без надмірностей, пропонованого модулями, або створити власну високорівневу абстракцію (подібно [» Doctrine MongoDB](https://github.com/doctrine/mongodb) для старого драйвера mongo). Майбутні бібліотеки можуть містити PHP-бібліотеку, призначену для адміністрування MongoDB, яка надає API для різних команд керування користувачами та командами ops. Наступна основна версія [» Doctrine MongoDB ODM](https://github.com/doctrine/mongodb-odm), швидше за все, також буде розміщено безпосередньо над модулями.

Хоча ми будемо продовжувати підтримувати і підтримувати старий драйвер mongo та його користувачів у найближчому майбутньому, ми запрошуємо всіх використовувати драйвер наступного покоління і розглядати його для будь-яких нових проектів у майбутньому.
